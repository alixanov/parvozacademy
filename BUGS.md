# Список ошибок проекта Parvoz Academy

> Дата анализа: 2026-06-11  
> Только реальные баги — без стилевых замечаний

---

## 🔴 КРИТИЧЕСКИЕ (безопасность / потеря данных)

### 1. Пароли хранятся в открытом виде
**Файл:** `server/src/models/User.model.js`, `server/src/modules/users/users.service.js`  
**Проблема:** Поле `passwordPlain` хранит пароль как обычный текст и отдаётся в admin-ответах.  
**Последствие:** При утечке базы — все пароли пользователей скомпрометированы.  
**Исправление:** Удалить поле `passwordPlain` из модели и сервиса.

---

### 2. Маршрут просмотра посещаемости — без авторизации
**Файл:** `server/src/modules/attendance/attendance.routes.js`  
**Проблема:** `GET /attendance/student/:studentId` не требует токена — любой посторонний может смотреть посещаемость любого ученика по ID.  
**Исправление:** Добавить `authenticate` middleware на этот маршрут.

---

### 3. Смена пароля без проверки старого пароля
**Файл:** `server/src/modules/users/users.service.js` (строки 90–98)  
**Проблема:** `updateUser()` позволяет сменить пароль без подтверждения текущего. Если токен украден — злоумышленник меняет пароль без знания старого.  
**Исправление:** При смене пароля требовать `currentPassword` и проверять через `bcrypt.compare`.

---

### 4. Ученик может подать заявку от имени другого ученика
**Файл:** `server/src/modules/enrollment/enrollment.controller.js`  
**Проблема:** Публичный маршрут `POST /enrollment-applications` принимает `student` ID из тела запроса без проверки владельца. Авторизованный ученик может передать чужой ID.  
**Исправление:** Если пользователь авторизован, принудительно ставить `student: req.user._id`, игнорируя тело.

---

### 5. Нет rate limiting на авторизацию
**Файл:** `server/src/modules/auth/auth.routes.js`  
**Проблема:** `POST /auth/login` и `POST /auth/register` без ограничений запросов — открыты для брутфорса паролей и спама аккаунтов.  
**Исправление:** Добавить `rateLimit` (например, 10 попыток / 15 минут) на login и register.

---

### 6. Нет транзакций при одобрении заявки на зачисление
**Файл:** `server/src/modules/enrollment/enrollment.service.js` (строки 92–229)  
**Проблема:** Одобрение создаёт пользователя → добавляет в группу → создаёт платёж → обновляет заявку. Если любой шаг падает, предыдущие уже выполнены — база становится несогласованной.  
**Последствие:** Пользователь создан, но не зачислен и без платежа.  
**Исправление:** Обернуть всё в MongoDB транзакцию (`session.withTransaction`).

---

## 🟠 СЕРЬЁЗНЫЕ (сломанная логика / неверное поведение)

### 7. Статус `'partial'` не существует в схеме Payment
**Файл:** `server/src/modules/payments/payments.service.js` (строка ~63), `server/src/models/Payment.model.js`  
**Проблема:** В коде используется `payment.status = 'partial'`, но в модели enum только `['debt', 'pending', 'paid']`. Mongoose выбросит ошибку валидации при подтверждении частичного платежа.  
**Исправление:** Добавить `'partial'` в enum модели или убрать логику частичных платежей.

---

### 8. Асинхронное обновление блокировки без `await`
**Файл:** `server/src/modules/groups/groups.service.js` (строка ~381)  
**Проблема:**
```js
GroupMember.findByIdAndUpdate(member._id, { paymentBlocked: shouldBlock }).exec();
// ← нет await, ошибка игнорируется
```
Если запрос к БД упадёт — в базе старый статус, в памяти новый. Студент может получить доступ, который ему закрыт.  
**Исправление:** Добавить `await` и обработку ошибки.

---

### 9. Сумма платежа из заявки, а не из группы
**Файл:** `server/src/modules/enrollment/enrollment.service.js` (строки ~186–202)  
**Проблема:** При одобрении создаётся платёж с `app.amount` (сумма на момент подачи заявки), а не `group.price.amount`. Если цена изменилась — платёж создаётся с неверной суммой.  
**Исправление:** Использовать `group.price.amount` при создании платежа.

---

### 10. Счётчик студентов курса считает дубликаты
**Файл:** `server/src/modules/groups/groups.service.js` (строки ~233–265)  
**Проблема:** При добавлении студента в группу выполняется `Course.$inc({ totalStudents: 1 })`. Если студент записан в 2 группы одного курса — счётчик увеличивается дважды.  
**Исправление:** Считать уникальных студентов через агрегацию, а не инкремент.

---

### 11. Уволенные/выпускники получают новые платёжные записи
**Файл:** `server/src/modules/groups/groups.service.js` (строки ~303–310)  
**Проблема:** При генерации платежей не фильтруются члены группы со статусом `'graduated'`, `'suspended'`, `'inactive'` — им тоже создаются долги.  
**Исправление:** Фильтровать только `members.filter(m => m.status === 'active')`.

---

### 12. Тихая ошибка при создании группы для пакета
**Файл:** `server/src/modules/packages/packages.service.js` (строки ~336–367)  
**Проблема:**
```js
} catch (groupErr) {
  console.error('[grantAccess] group auto-create failed:', groupErr.message);
  // продолжает без группы
}
```
Студент получает доступ к пакету, но без группы — не может проходить контент.  
**Исправление:** Прокидывать ошибку выше (`throw groupErr`), откатывать весь `grantAccess`.

---

### 13. Посещаемость не проверяет принадлежность студентов к группе
**Файл:** `server/src/modules/attendance/attendance.service.js`  
**Проблема:** При отметке посещаемости не проверяется, что студенты в `records[]` реально состоят в этой группе.  
**Последствие:** Учитель может отметить посещаемость случайных пользователей.  
**Исправление:** Проверить каждый `student` через `GroupMember.exists({ group, student })`.

---

## 🟡 СРЕДНИЕ (некорректное поведение / UX)

### 14. RTK Query не инвалидирует кэш `User` при одобрении заявки
**Файл:** `client/src/features/enrollment/enrollmentApi.js`  
**Проблема:** `approveEnrollment` инвалидирует `['Enrollment', 'Group', 'Payment']`, но не `'User'`. Новый студент не появляется в списке студентов до перезагрузки страницы.  
**Исправление:** Добавить `'User'` и `'Course'` в `invalidatesTags`.

---

### 15. Неполная валидация телефона на фронтенде
**Файл:** `client/src/pages/public/Enroll/index.jsx`  
**Проблема:** Форма принимает менее 9 цифр и отправляет на сервер вида `+998123` (неполный номер). Сервер выдаёт неинформативную ошибку.  
**Исправление:** Блокировать отправку если `form.phone.length !== 9`.

---

### 16. Загрузка файла не обрабатывает не-JSON ответы
**Файл:** `client/src/pages/public/Enroll/index.jsx` (строки ~225–244)  
**Проблема:** `res.json()` вызывается без проверки `res.ok`. При 500-ошибке сервер вернёт HTML, `res.json()` упадёт с непонятным сообщением.  
**Исправление:**
```js
if (!res.ok) throw new Error(`Upload error: ${res.status}`);
const json = await res.json();
```

---

### 17. Socket.IO — нет проверки JWT при подключении
**Файл:** `server/src/config/socket.js`  
**Проблема:** При handshake читается `socket.handshake.auth.userId` без верификации токена. Любой может передать чужой `userId` и получать чужие уведомления.  
**Исправление:** Верифицировать `socket.handshake.auth.token` через `verifyAccess()`, брать `userId` из токена.

---

### 18. Дата блокировки зависит от часового пояса сервера
**Файл:** `server/src/modules/groups/groups.service.js`  
**Проблема:** Сравнение `graceDeadline <= now` работает корректно только если сервер и клиент в одном часовом поясе. Railway сервера работают в UTC, Узбекистан — UTC+5.  
**Исправление:** Все сравнения дат делать через Unix timestamp (`.getTime()`).

---

### 19. Дублирующийся `passwordPlain` при обновлении пользователя
**Файл:** `server/src/modules/users/users.service.js`  
**Проблема:** При смене пароля через admin-API `passwordPlain` обновляется, но если пользователь меняет пароль через `PATCH /users/me` — `passwordPlain` остаётся старым.  
**Исправление:** Удалить поле (см. баг #1).
